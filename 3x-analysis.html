一、整體結論

經比對三個 HTM 搜證樣本後確認：

  • 三個網頁的 HTML 與 JavaScript 架構高度相似，相似度約達 97%
  • 核心程式邏輯完全一致
  • 差異主要集中於 token、追蹤參數與識別碼
  • 屬於同一套導流模板大量複製部署的典型模式
🔑 關鍵結論 三個網域共用同一組 Monetag Publisher Key，歸屬同一變現帳戶，為本案金流追查之核心依據。
二、共同架構
2.1　基本 HTML 結構

三者皆採用極簡 HTML，幾乎無正常網站內容，所有功能均集中於 JavaScript：

<!doctype html><html lang="en"><head><meta charset="utf-8"/></head> <body><form action="" id="submit-form"></form> <script> /* 導流邏輯 */ </script></body></html>
  • 無可視 UI，頁面對使用者完全透明
  • 使用隱藏 <form> 作為資料提交媒介
  • 功能集中於 JavaScript，頁面載入即執行
2.2　Hidden Form 機制

程式動態建立 hidden input 欄位（createElement('input')），用途如下：

  • 蒐集裝置資訊
  • 建立 session token
  • 背景送出資料至 C2
  • 自動 submit 觸發導流
三、裝置資訊蒐集

每次頁面載入，JS 自動蒐集以下裝置資訊，用於判斷真人 / 爬蟲並決定分流策略：

蒐集欄位用途說明
window.innerWidth / Height瀏覽器視窗寬高
screen.width / height螢幕實體解析度
devicePixelRatio裝置像素比（判斷高解析螢幕）
documentElement.clientWidth頁面可視寬高
window.self vs window.top判斷是否在 iframe 內執行（fr 欄位）

偵測目的：

  • 判斷訪客為真人或爬蟲 / 安全分析工具
  • 分流受害者至不同落地頁
  • 規避自動化安全分析
四、Token 與 Key 差異分析
4.1　fstu — 每 session 唯一訪客識別碼

三個網域的 fstu 值均不同，格式為 UUID v7：

網域fstu 值
zixbin.info 英明019e1a3e-569a-7c71-a07c-00c21ee578f1
cuaeww.info 英明019e1a3f-8caa-75c2-938c-ad5d521f2339
zixbin.info ptt019e1a40-3bbf-7bef-bf45-36c9aae8a7e1
UUID v7 說明 高位 48 bits 內嵌 Unix 毫秒時間戳，代表訪客抵達頁面的精確時刻。此欄位每個 session 獨立生成，不具跨 session 歸因能力，不作為身份識別依據使用。

五、運作原理

完整攻擊鏈流程如下：

  1. 受害者透過 Google 搜尋觸發 SEO 汙染結果，點擊進入目標網域
  2. 頁面載入極簡 HTML，立即執行 JavaScript 導流腳本
  3. JS 蒐集裝置資訊，背景送出至 C2（protrafficinspector.com/stats）取得 uuid cookie
  4. TDS（流量分發系統）依裝置特徵及 uuid 判斷，決定導向目標
  5. 受害者被導向最終落地頁

可能的落地頁類型：

  • 惡意軟體下載頁
  • 偽防毒 / 勒索軟體誘騙頁
  • 非法色情網站
  • 詐騙交易頁
  • Affiliate 聯盟行銷落地頁
六、鑑識推論
  1. 三個頁面來自同一套程式模板，批次生成部署
  2. 由同一導流平台（Monetag SmartLink）統一管理，共用帳號
  3. 不同 token（fstu、ps1）用於後端流量識別與計費，不影響帳號歸因
  4. 網域本身僅是流量入口，真正控制核心位於後端 TDS 系統
  5. Publisher Key 跨域共用，確認三個網域歸屬同一 JX 變現帳戶
七、結論

本案技術特徵高度符合以下典型攻擊型態：

  • SEO 汙染（Search Engine Poisoning）
  • TDS 流量導向（Traffic Distribution System）
  • 惡意廣告聯盟變現（Malvertising / Affiliate Fraud）
  • Malware redirect chain

  • 作為聲請凍結相關廣告收益帳戶之依據
  • 提供 FBI / DOJ / FTC 跨境調查之技術歸因基礎
8.2　新發現：第二組金流 Key（reverse urlPath 機制）

三個 HTM 檔案的 reverse urlPath 中，解碼出此前未曾出現的 key：

欄位數值
reverse key10afaec35404ffd5edbad0131bb38ee6
submetric（廣告位 ID）28656948
reverse urlPath 機制技術說明

此機制在 popunder 廣告觸發時運作，透過 window.history.pushState 在不導離頁面的情況下，將瀏覽器顯示的 URL 改寫為含有此 key 的路徑：

/api/users?token=L3VjNncwcmVrcnM_...
  • 目的：讓廣告後端統計系統記錄本次流量事件並計費
  • 同時規避使用者察覺實際導流行為
  • submetric（28656948）代表具體廣告版位或流量計費節點
⚠ 兩組 key 歸屬待查 若兩組 key 歸屬同一 Publisher 帳號 → 歸因更加確定。
若歸屬不同帳號 → 可能揭露 JX 在 Monetag 平台上的多帳號結構。
建議將第二組 key 附入 Monetag 法律請求補充說明中，一併追查。

九、st1 時間戳記鑑識說明
⚠ 重要區分 st1 記錄的是搜證人員的瀏覽時間，不是 JX 的操作時間，兩者不應混淆。

st1 欄位為奈秒級 Unix 時間戳，由 client-side JavaScript 在訪客載入頁面時即時生成，記錄搜證當下的瀏覽時刻。

9.2　時間戳記之鑑識意義
  1. 確認三個頁面於搜證當日（2026-05-12 上午 11:32–11:34）仍在線上活躍運行（live domain），可作為「頁面存活」之時間佐證。
  2. fstu UUID v7 亦內嵌搜證當下的時間戳，兩項互相印證。
  3. 此時間與 JX 操作行為無關，不得作為 JX 批次自動化之時間指紋引用。
⚠ JX 操作時間指紋應引用 WHOIS 記錄中發現之 16:30–16:32 批次域名註冊時間（台灣時間）。
請嚴格區分「搜證時間」與「JX 操作時間」，避免誤導審閱者。
